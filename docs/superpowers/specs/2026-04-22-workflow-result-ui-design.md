# 工作流结果卡片 UI 设计

## 背景

当前聊天页中的 `新闻推送` 和 `个税报表` 结果都沿用通用 Markdown 气泡渲染。内容虽然可读，但存在几个明显问题：

- 结构化结果被排成普通长文，信息层级弱
- 标题、元信息、摘要、操作按钮混在同一文本流里，视觉节奏不稳定
- `个税报表` 的统计结果像“列表 + 两个链接”，缺少结果面板感
- `新闻推送` 的多条新闻连续平铺，阅读压力大，操作入口不聚焦
- 同一聊天线程里，工作流结果和普通回答几乎没有展示语义差异

本次设计目标不是重做聊天整体 UI，而是在现有会话壳子内，为工作流结果增加专属展示层，使 `新闻推送` 更像编辑简报，`个税报表` 更像工作台结果卡，同时保持统一的视觉语言。

## 目标

- 为 `新闻推送` 和 `个税报表` 提供专属结果卡片，不再完全依赖通用 Markdown 排版
- 保留现有聊天左右布局、头像区、工作流标签和更多操作菜单
- 统一颜色、圆角、按钮、边框和间距系统，避免两个工作流像来自两套界面
- 提升结果扫描效率，让用户一眼看懂“这次工作流做了什么”和“下一步操作是什么”
- 在不改动后端协议的前提下，优先利用现有 `message.content` 和 `workflowInvocation` 做前端识别与渲染

## 非目标

- 不重构整页聊天布局
- 不重写普通 AI 回答的 Markdown 渲染体验
- 不要求后端立刻改成结构化返回
- 不处理除 `新闻推送`、`个税报表` 之外的所有工作流展示

## 方案选择

### 方案 A：工作流专属结果卡片

在前端根据 `workflowInvocation` 和内容结构识别工作流消息，命中后走专门组件。

优点：

- 视觉提升最大
- 可以为 `新闻推送` 和 `个税报表` 做不同但统一的版式
- 不需要大改后端返回格式

缺点：

- 需要前端增加一层解析逻辑
- 对消息格式有一定依赖，需要容错

### 方案 B：继续使用 Markdown，仅增强样式

只优化 `MarkdownRenderer` 的标题、列表、链接、按钮和块间距。

优点：

- 改动小
- 风险低

缺点：

- 只能把“难看”变成“稍微顺眼”
- 解决不了结构化结果缺少专属布局的问题

### 方案 C：后端直接返回结构化结果

工作流执行接口直接返回 stats、articles、actions 等字段，前端只负责渲染。

优点：

- 展示最稳，长期维护最好

缺点：

- 改动范围更大
- 不适合这次先快速解决现有界面问题

### 结论

采用方案 A。当前阶段先在前端增加专属结果卡片层，保留未来向方案 C 过渡的空间。

## 视觉方向

采用折中方案：

- `新闻推送` 偏编辑简报感，强调标题层级、阅读节奏和单条新闻卡片化
- `个税报表` 偏专业工作台感，强调结果状态、关键统计和明确的操作区
- 二者共享一套基础设计语言：柔和浅底、统一边框、细阴影、克制按钮、相同圆角体系

整体风格基调：

- 不使用大面积艳色块
- 状态通过边框、浅背景和局部强调色表达
- 标题使用当前项目已有的编辑风格字体体系，正文保持稳定可读
- 操作按钮主次明确，避免多个“像链接一样”的按钮并排

## 信息架构

### 个税报表

个税报表结果卡结构：

1. 顶部状态区
   - 成功状态标签
   - 标题：`合并数据成功`
   - 可选补充文案：如“已生成标准个税申报表”
2. 中部统计区
   - `识别发放记录表`
   - `识别结算发放表`
   - `合并人员数量`
3. 底部操作区
   - 主按钮：`下载合并结果`
   - 次按钮：`查看 output 文件夹`

展示原则：

- 统计项不再使用项目符号列表，而是用等宽小指标块
- 按钮区与统计区分层，避免所有元素堆在同一段文字里
- 如果后续增加异常行数、跳过条数，也放在统计区扩展，不回退到长文

### 新闻推送

新闻推送结果结构：

1. 顶部简报头
   - 标题：`新闻推送简报`
   - 本次爬虫关键词
   - 用户关注点摘要
2. 新闻列表
   - 每条新闻独立为 article card
   - 标题单独一行，强调可扫读性
   - 元信息一行：来源、发布时间、关键词
   - 摘要作为正文段落或短列表
   - 操作区单独成行：`保存`、`引用`

展示原则：

- 一条新闻就是一个独立视觉单元，不能再整篇串联成单列长文
- 元信息和摘要分层，操作区固定在条目底部
- 保留 Markdown 中的链接，但工作流操作按钮优先以统一按钮样式呈现

## 组件设计

新增前端渲染分层：

1. `ConversationItem`
   - 保留头像、时间、工作流标签、更多操作
   - 在 assistant message 内增加“工作流结果判定”
2. `WorkflowResultCard`
   - 作为统一入口组件，根据类型分发
3. `TaxReportResultCard`
   - 渲染个税报表结果卡
4. `NewsDigestResultCard`
   - 渲染新闻推送简报卡

通用 Markdown 继续作为兜底：

- 未命中的 assistant message 仍走 `MarkdownRenderer`
- 工作流结果解析失败时回退到 `MarkdownRenderer`

## 内容识别与解析

第一阶段不依赖后端新增结构化字段，前端从已有消息信息中提取：

### 识别条件

- 若 `message.workflowInvocation === '/个税报表'`，优先尝试个税卡片解析
- 若 `message.workflowInvocation === '/新闻推送'`，优先尝试新闻简报解析

### 个税报表解析规则

从 `message.content` 中识别：

- 标题成功态文案
- 三项统计值
- `downloadUrl` 或 action link
- `查看 output 文件夹` action

若统计值解析不全，但存在成功标题和动作区，仍尽量展示卡片；缺失字段显示为 `--` 而不是回退零散段落。

### 新闻推送解析规则

从 `message.content` 中识别：

- 顶部“本次爬虫关键词”
- “对话输入关注点”
- 文章编号段落
- 文章元信息列表
- `保存`、`引用` 动作

解析失败容错策略：

- 如果能识别出多条文章，就渲染 article card
- 若内容结构被破坏，只保留简报头，正文降级为 Markdown
- 如果完全无法解析，则整体回退 `MarkdownRenderer`

## 样式系统

### 共享样式

- 卡片底色：暖白或极浅灰白
- 边框：低对比浅灰边框
- 阴影：小而软，不做浮夸投影
- 圆角：比普通消息泡略大，但不脱离当前系统
- 主文字：深 slate
- 辅助文字：降低对比度但保持可读

### 个税报表卡

- 状态标签使用浅蓝或浅青背景，不做高饱和成功绿
- 统计块使用规则栅格排列，数字强化、标签减弱
- 主按钮强调下载，次按钮退一级

### 新闻简报卡

- 简报头用轻量分隔，不做厚重 banner
- 每篇文章卡片边界清晰，但间距比个税结果更松
- 标题更像编辑标题，摘要行高偏舒展
- 操作按钮靠近条目底部，避免和正文混杂

## 交互细节

- 所有按钮点击热区维持至少 44px 高度
- hover 和 active 反馈统一，避免一个像网页链接、一个像系统按钮
- `保存`、`引用`、`下载结果`、`查看 output 文件夹` 均采用统一按钮语义
- 工作流标签继续显示在头像旁，作为消息级上下文，不移入卡片内部

## 响应式与兼容性

- 桌面端优先保证阅读和统计信息横向布局
- 窄屏下：
  - 个税统计块从三列降为一列或两列
  - 新闻 article card 保持单列
  - 按钮区允许换行，但主次顺序不变
- 不引入依赖 hover 才可见的重要信息

## 实现步骤

1. 在 `src/components` 中新增工作流结果组件
2. 在 `ConversationItem.tsx` 中插入工作流结果识别与分发逻辑
3. 抽出简单解析函数，将 Markdown 内容转成前端视图模型
4. 为卡片和操作按钮补充专用样式
5. 保留 `MarkdownRenderer` 作为回退路径

## 测试策略

### 单元测试

- 个税报表内容解析：成功、缺字段、动作缺失、回退
- 新闻简报内容解析：多条新闻、部分字段缺失、结构损坏回退

### 组件验证

- `ConversationItem` 在两类工作流消息下命中专属卡片
- 普通 assistant 消息仍走原 Markdown 渲染

### 手动验证

- `/个税报表` 成功结果展示
- `新闻推送` 多条新闻展示
- 按钮点击行为保持原有逻辑
- 小屏宽度下无明显挤压和横向滚动

## 风险与后续演进

### 当前风险

- 依赖字符串解析，未来若后端文案格式调整，前端识别可能失效
- 新闻简报内容结构如果差异过大，第一版解析需保守降级

### 对应缓解

- 解析层与展示层分离
- 失败时统一回退到 `MarkdownRenderer`
- 后续可逐步把工作流返回升级为结构化 artifacts

## 后续可扩展项

- 为工作流结果增加轻量入场动画
- 为新闻卡片增加折叠/展开摘要
- 为个税卡片增加“本次处理文件数”和异常提醒
- 把工作流结果卡片能力扩展到更多内置工作流
